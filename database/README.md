# قاعدة بيانات النظام المصرفي الآمن

## 1. مقدمة

يحتوي هذا المجلد على الملفات المتعلقة بتصميم قاعدة بيانات مشروع النظام المصرفي الآمن.

يعتمد المشروع على قاعدة بيانات SQLite أثناء التطوير والاختبار، ويتم التعامل معها باستخدام:

Flask-SQLAlchemy
تُنشأ قاعدة البيانات الفعلية داخل مجلد:

backend/instance/
ولا تُحفظ قاعدة البيانات الحقيقية داخل هذا المجلد.

---

## 2. وظيفة مجلد Database

يستخدم هذا المجلد لحفظ:

- وصف قاعدة البيانات.
- مخطط الجداول.
- ملفات SQL المساعدة.
- أوامر إنشاء الجداول.
- بيانات تجريبية غير حساسة.
- تعليمات النسخ الاحتياطي.
- تعليمات نقل النظام إلى PostgreSQL مستقبلًا.

---

## 3. قاعدة البيانات المستخدمة

يستخدم المشروع:

SQLite
وهي قاعدة بيانات مناسبة للمشاريع التعليمية والتطبيقات المحلية.

مميزات SQLite:

- لا تحتاج إلى خادم مستقل.
- سهلة الإعداد.
- تخزن البيانات داخل ملف واحد.
- متوافقة مع Flask-SQLAlchemy.
- مناسبة للاختبارات.
- سهلة النقل والنسخ الاحتياطي.

---

## 4. موقع قاعدة البيانات الفعلية

يتم إنشاء قاعدة البيانات داخل:

backend/instance/
وقد يكون اسم الملف:

banking.db
أو اسمًا مشابهًا حسب إعدادات المشروع.

مثال للمسار:

backend/instance/banking.db
---

## 5. حماية ملف قاعدة البيانات

لا يجب رفع قاعدة البيانات الفعلية إلى GitHub؛ لأنها قد تحتوي على:

- أسماء المستخدمين.
- عناوين البريد الإلكتروني.
- كلمات المرور المشفرة.
- أرقام الحسابات.
- الأرصدة.
- سجلات العمليات المالية.

لذلك تمت إضافة الملفات التالية إلى .gitignore:

backend/instance/
*.db

*.sqlite3

## 6. جداول قاعدة البيانات

تتكون قاعدة البيانات من ثلاثة جداول رئيسية:

users

transactions
---

## 7. جدول المستخدمين

اسم الجدول:

users
يخزن بيانات المستخدمين والمديرين.

### الحقول

| الحقل | نوع البيانات | الوصف |
|---|---|---|
| id | Integer | الرقم الأساسي للمستخدم |
| username | String | اسم المستخدم |
| email | String | البريد الإلكتروني |
| password_hash | String | كلمة المرور المشفرة |
| role | String | صلاحية المستخدم |
| is_active | Boolean | حالة المستخدم |
| created_at | DateTime | تاريخ إنشاء الحساب |

### القيود

- id مفتاح أساسي.
- username يجب أن يكون فريدًا.
- email يجب أن يكون فريدًا.
- كلمة المرور الأصلية لا تُحفظ داخل الجدول.
- القيمة الافتراضية للصلاحية هي customer.
- القيمة الافتراضية للحالة هي True.

---

## 8. جدول الحسابات البنكية

اسم الجدول:

accounts
يخزن الحساب البنكي المرتبط بالمستخدم.

### الحقول

| الحقل | نوع البيانات | الوصف |
|---|---|---|
| id | Integer | الرقم الداخلي للحساب |
| account_number | String | رقم الحساب البنكي |
| balance | Numeric | الرصيد الحالي |
| user_id | Integer | رقم صاحب الحساب |

### القيود

- id مفتاح أساسي.
- account_number يجب أن يكون فريدًا.
- user_id مفتاح أجنبي.
- كل حساب مرتبط بمستخدم.
- كل مستخدم يمتلك حسابًا واحدًا.

---

## 9. جدول العمليات المالية

اسم الجدول:

transactions
يخزن التحويلات المالية الناجحة.

### الحقول

| الحقل | نوع البيانات | الوصف |
|---|---|---|
| id | Integer | رقم العملية |
| sender_account_id | Integer | حساب المرسل |
| receiver_account_id | Integer | حساب المستلم |
| amount | Numeric | مبلغ التحويل |
| created_at | DateTime | تاريخ ووقت العملية |

### القيود

- id مفتاح أساسي.
- sender_account_id مفتاح أجنبي.
- receiver_account_id مفتاح أجنبي.
- يجب أن يكون المبلغ أكبر من صفر.
- لا يتم تسجيل العمليات المرفوضة.

---

## 10. العلاقات بين الجداول

العلاقة بين المستخدم والحساب:

User 1 -------- 1 Account
كل مستخدم يمتلك حسابًا بنكيًا واحدًا.

العلاقة بين الحساب والعمليات:

Account 1 -------- Many Transactions
الحساب الواحد يمكن أن:

- يرسل عدة عمليات.
- يستقبل عدة عمليات.

## 14. تنفيذ التحويل المالي

عند تنفيذ تحويل ناجح تتم العمليات التالية:

1. قراءة حساب المرسل.
2. قراءة حساب المستلم.
3. التحقق من صحة المبلغ.
4. التحقق من توفر الرصيد.
5. خصم المبلغ من المرسل.
6. إضافة المبلغ إلى المستلم.
7. إنشاء سجل في جدول العمليات.
8. حفظ التغييرات.

عند النجاح يستخدم النظام:

python
db.session.commit()

عند حدوث خطأ يستخدم:

python
db.session.rollback()

## 15. أهمية استخدام Decimal

لا يُفضل استخدام:

python
float

في العمليات المالية؛ لأنه قد ينتج أخطاء تقريب.

يستخدم المشروع:

python
Decimal

للتعامل مع المبالغ المالية بدقة.

مثال:

python
from decimal import Decimal

amount = Decimal("100.50")

## 16. قاعدة بيانات الاختبارات

تستخدم الاختبارات قاعدة بيانات مستقلة عن قاعدة البيانات الحقيقية.

يتم إعدادها داخل:

text
tests/conftest.py

في بداية الاختبار يتم:

1. إنشاء تطبيق للاختبار.
2. إنشاء قاعدة بيانات مؤقتة.
3. إنشاء الجداول.
4. تنفيذ الاختبار.
5. حذف الجداول بعد انتهاء الاختبار.

وبذلك لا تؤثر الاختبارات على بيانات المشروع الحقيقية.

---

## 17. عدم تعديل قاعدة البيانات يدويًا

يُفضل عدم تعديل ملف SQLite يدويًا أثناء تشغيل التطبيق.

يتم تنفيذ العمليات من خلال:

text
Flask-SQLAlchemy

وذلك لتقليل مخاطر:

- تلف البيانات.
- حذف سجلات مهمة.
- تغيير الرصيد بصورة غير صحيحة.
- كسر العلاقات بين الجداول.
- إدخال بيانات غير متوافقة مع النماذج.

---

## 18. النسخ الاحتياطي

لإنشاء نسخة احتياطية أثناء التطوير:

1. أوقف تشغيل التطبيق.
2. افتح المجلد:

text
backend/instance/

3. انسخ ملف قاعدة البيانات.
4. احفظ النسخة في مكان آمن خارج مستودع GitHub.

مثال لاسم النسخة:

text
banking_backup_2026-07-12.db

لا ترفع النسخة الاحتياطية إلى مستودع عام.

---

## 19. استعادة النسخة الاحتياطية

لاستعادة النسخة:

1. أوقف تشغيل التطبيق.
2. احتفظ بنسخة من قاعدة البيانات الحالية.
3. ضع ملف النسخة الاحتياطية داخل:

text
backend/instance/

4. تأكد من تطابق اسم الملف مع إعدادات الاتصال.
5. أعد تشغيل التطبيق.

---

## 20. استخدام قاعدة بيانات إنتاجية

SQLite مناسبة للمشروع التعليمي، لكن عند نشر نظام حقيقي يفضل استخدام:

text
PostgreSQL

أو:

text
MySQL

توفر قواعد البيانات الإنتاجية:

- دعم عدد أكبر من المستخدمين.
- إدارة أفضل للاتصالات المتزامنة.
- نسخًا احتياطية متقدمة.
- صلاحيات أكثر دقة.
- أداء أفضل.
- حماية أكبر.
- دعم المعاملات بصورة أوسع.

سيحتوي مجلد `database` على:
database/
├── README.md
└── schema.sql
- `README.md: شرح قاعدة البيانات.

## 24. الخاتمة

تستخدم قاعدة بيانات النظام المصرفي الآمن ثلاثة جداول رئيسية لحفظ المستخدمين والحسابات والتحويلات المالية.

تم تصميم العلاقات بطريقة تسمح بربط كل مستخدم بحسابه، وتسجيل العمليات المرسلة والمستلمة.

كما يعتمد النظام على SQLAlchemy وعمليات commit وrollback` لضمان سلامة البيانات أثناء تنفيذ التحويلات.
