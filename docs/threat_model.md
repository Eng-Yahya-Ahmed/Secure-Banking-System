# نموذج التهديدات الأمنية للمشروع

## 1. مقدمة

يوثق هذا الملف نموذج التهديدات الأمنية الخاص بمشروع Secure Banking System.

يهدف نموذج التهديدات إلى تحديد الأصول الحساسة، ونقاط الدخول، وحدود الثقة، والتهديدات المحتملة، والضوابط الأمنية المطبقة، والمخاطر المتبقية.

تم استخدام منهجية STRIDE لتحليل التهديدات.

---

## 2. نطاق التحليل

يشمل هذا التحليل الأجزاء التالية:

- تسجيل المستخدمين.
- تسجيل الدخول والخروج.
- إدارة جلسات المستخدمين.
- الحسابات البنكية.
- التحويلات المالية.
- سجل المعاملات.
- الملف الشخصي.
- تغيير كلمة المرور.
- لوحة المدير.
- قاعدة بيانات SQLite.
- تطبيق Flask.
- تشغيل التطبيق باستخدام Docker.
- الصفحات العامة والصفحات المحمية.

لا يشمل هذا التحليل:

- البنية التحتية السحابية.
- بوابات الدفع الخارجية.
- تطبيقات الهاتف المحمول.
- خدمات البريد الإلكتروني.
- أنظمة المصادقة الخارجية.
- API Gateway.
- Microservices.

## 6. حدود الثقة

### الحد الأول: المستخدم والمتصفح

البيانات القادمة من المتصفح غير موثوقة، حتى لو كانت النماذج تحتوي على تحقق في HTML.

يجب التحقق من جميع البيانات داخل الخادم.

### الحد الثاني: تطبيق Flask وقاعدة البيانات

التطبيق هو الجهة الوحيدة المسموح لها بالتعامل المباشر مع قاعدة البيانات.

يجب عدم السماح للمستخدم بكتابة استعلامات SQL مباشرة.

### الحد الثالث: التطبيق والموارد الخارجية

يتم تحميل Bootstrap من CDN خارجي.

تمت إضافة Subresource Integrity للتحقق من سلامة الملفات الخارجية.

### الحد الرابع: الجهاز المضيف وحاوية Docker

يعمل التطبيق داخل حاوية Docker، بينما يتم الوصول إليه من الجهاز المضيف عبر منفذ محلي.

## 7. تحليل STRIDE

## 7.1 انتحال الهوية — Spoofing

### سيناريوهات التهديد

- استخدام كلمة مرور مسروقة.
- تخمين كلمة المرور.
- سرقة Session Cookie.
- محاولة تسجيل الدخول باسم مستخدم آخر.

### الضوابط الأمنية

- تخزين كلمات المرور باستخدام Bcrypt.
- استخدام Flask-Login لإدارة الجلسات.
- تفعيل HttpOnly Cookies.
- استخدام SameSite Cookies.
- تطبيق Rate Limiting على تسجيل الدخول.
- التحقق من كلمة المرور المخزنة باستخدام Hash.
- حماية الصفحات باستخدام login_required.

### المخاطر المتبقية

- عدم وجود مصادقة ثنائية.
- عدم وجود قفل مؤقت للحساب بعد المحاولات الفاشلة.
- عدم وجود تنبيه عند تسجيل الدخول من جهاز جديد.

---

## 7.2 التلاعب — Tampering

### سيناريوهات التهديد

- تعديل قيمة المبلغ قبل إرسال الطلب.
- تغيير رقم حساب المستلم.
- محاولة تعديل الرصيد مباشرة.
- التلاعب بمعاملات HTTP.
- محاولة تعديل ملفات JavaScript أو CSS الخارجية.

### الضوابط الأمنية

- التحقق من المبلغ داخل الخادم.
- رفض المبالغ السالبة أو الصفرية.
- رفض المبالغ التي تتجاوز الرصيد.
- منع أكثر من منزلتين عشريتين.
- التحقق من وجود حساب المستلم.
- منع التحويل إلى الحساب نفسه.
- استخدام SQLAlchemy ORM.
- استخدام CSRF Token.
- استخدام Subresource Integrity.
- استخدام معاملات قاعدة البيانات وRollback عند الخطأ.

### المخاطر المتبقية
- SQLite محدودة في حالات التزامن العالية.
- لا يوجد نظام متقدم لقفل الصفوف المالية.
- لا يوجد توقيع رقمي مستقل لكل معاملة.

---

## 7.3 إنكار تنفيذ العملية — Repudiation

### سيناريوهات التهديد

- إنكار المستخدم أنه نفذ تحويلًا.
- إنكار المدير أنه نفذ إجراءً إداريًا.
- عدم وجود دليل على محاولة دخول فاشلة.
- عدم وجود سجل مستقل لتغيير كلمة المرور.

### الضوابط الأمنية الحالية

- تخزين عمليات التحويل داخل جدول Transaction.
- تخزين المرسل والمستلم والمبلغ والتاريخ.
- إمكانية مراجعة سجل المعاملات.

### المخاطر المتبقية

- لا يوجد Audit Logging شامل.
- لا يتم تسجيل عنوان IP.
- لا يتم تسجيل محاولات الدخول الفاشلة.
- لا يتم تسجيل إجراءات المدير بشكل مستقل.

### التحسين المقترح

إضافة سجل تدقيق أمني يسجل:

- تسجيل الدخول الناجح والفاشل.
- تغيير كلمة المرور.
- التحويلات الناجحة والمرفوضة.
- دخول لوحة الإدارة.
- الإجراءات الإدارية.
- عنوان IP.
- وقت الحدث.

---

## 7.4 كشف المعلومات — Information Disclosure

### سيناريوهات التهديد

- كشف كلمات المرور.
- كشف Session Cookie.
- كشف معلومات إصدار الخادم.
- عرض رسائل أخطاء داخلية.
- وصول مستخدم إلى بيانات مستخدم آخر.
- كشف المفتاح السري داخل GitHub.

### الضوابط الأمنية

- عدم تخزين كلمات المرور الأصلية.
- استخدام Bcrypt.
- استخدام HttpOnly Cookies.
- تقليل معلومات إصدار الخادم.
- إنشاء صفحات أخطاء مخصصة.
- استخدام Role-Based Access Control.
- قراءة SECRET_KEY من متغير بيئة.
- عدم رفع ملفات قاعدة البيانات الحساسة.
- استخدام Referrer-Policy.
- استخدام Content-Security-Policy.

### المخاطر المتبقية

- التطبيق يعمل محليًا دون HTTPS.
- قاعدة البيانات غير مشفرة بالكامل.
- لا توجد إدارة مركزية للأسرار.

---

## 7.5 حجب الخدمة — Denial of Service

### سيناريوهات التهديد

- إرسال عدد كبير من طلبات تسجيل الدخول.
- إرسال طلبات تحويل متكررة.
- إغراق الخادم بطلبات HTTP.
- استهلاك موارد التطبيق أو قاعدة البيانات.

### الضوابط الأمنية

- استخدام Flask-Limiter.
- تحديد عدد محاولات تسجيل الدخول.
- تحديد عدد طلبات التحويل.
- صفحة خطأ 429.
- تشغيل التطبيق داخل Docker.
- التحقق من حجم وصحة البيانات المدخلة.

### المخاطر المتبقية

- لا يوجد Web Application Firewall.
- لا توجد حماية موزعة من DDoS.
- لا توجد مراقبة مركزية للموارد.
- التطبيق يستخدم خادم Flask التطويري.

---

## 7.6 رفع الصلاحيات — Elevation of Privilege

### سيناريوهات التهديد

- محاولة عميل الوصول إلى لوحة المدير.
- تعديل قيمة الدور داخل الطلب.
- استخدام رابط إداري مباشرة.
- استغلال خطأ برمجي للوصول إلى وظائف غير مصرح بها.

### الضوابط الأمنية

- استخدام أدوار customer وadmin.
- التحقق من دور المستخدم داخل الخادم.
- حماية المسارات باستخدام login_required.
- إرجاع 403 عند عدم وجود صلاحية.
- عدم الاعتماد على إخفاء الرابط فقط.
- استخدام Flask-Login لمعرفة المستخدم الحالي.

### المخاطر المتبقية

- لا يوجد نظام صلاحيات أكثر تفصيلًا.
- لا يوجد Audit Log للإجراءات الإدارية.
- لا توجد مراجعة دورية لصلاحيات المستخدمين.

## 12. الخلاصة

يوضح نموذج التهديدات أن المشروع يعالج مجموعة مهمة من مخاطر تطبيقات الويب، مثل انتحال الهوية، والتلاعب، وكشف المعلومات، وحجب الخدمة، ورفع الصلاحيات.

تم تطبيق عدة ضوابط أمنية ضمن مراحل S-SDLC، كما تم تنفيذ اختبارات SAST وDAST واختبارات برمجية تلقائية.

تظل بعض المخاطر مرتبطة بكون النظام نموذجًا أكاديميًا يعمل محليًا، ويمكن معالجتها مستقبلًا عند الانتقال إلى بيئة إنتاج.
