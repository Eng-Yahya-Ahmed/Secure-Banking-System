# التحسينات المستقبلية المقترحة

## 1. مقدمة

يوثق هذا الملف التحسينات المستقبلية المقترحة لمشروع Secure Banking System.

المشروع الحالي يحقق المتطلبات الأكاديمية الأساسية، ويطبق عددًا من ضوابط الأمان واختبارات SAST وDAST والاختبارات البرمجية والتشغيل باستخدام Docker.

تهدف التحسينات التالية إلى نقل المشروع من نموذج أكاديمي محلي إلى نظام أكثر قربًا من بيئة الإنتاج، مع تحسين الأمان والاعتمادية وقابلية التوسع وتجربة المستخدم.

---

## 2. الانتقال من SQLite إلى PostgreSQL

تُعد SQLite مناسبة للتطوير والتعليم والتجارب المحلية، لكنها ليست الخيار الأفضل لنظام مصرفي إنتاجي يتعامل مع عدد كبير من المستخدمين والعمليات المتزامنة.

### التحسين المقترح

الانتقال إلى:

```text
PostgreSQL
```

### الفوائد

- دعم أفضل للعمليات المتزامنة.
- دعم قوي للمعاملات.
- تحسين التحكم في صلاحيات قاعدة البيانات.
- سهولة إنشاء النسخ الاحتياطية والاستعادة.
- ملاءمة أكبر للأنظمة متعددة المستخدمين.
- دعم أفضل للتوسع.
- تقليل مخاطر تعارض تحديث الأرصدة.

### متطلبات التنفيذ

- تحديث إعداد `SQLALCHEMY_DATABASE_URI`.
- إنشاء مستخدم وقاعدة بيانات خاصة بالتطبيق.
- استخدام متغيرات البيئة لحفظ بيانات الاتصال.
- تنفيذ Migrations باستخدام Alembic أو Flask-Migrate.
- اختبار نقل البيانات من SQLite.
- تشغيل جميع الاختبارات بعد الانتقال.

## 4. إضافة المصادقة الثنائية MFA

تعتمد النسخة الحالية على اسم المستخدم وكلمة المرور.

### التحسين المقترح

إضافة عامل مصادقة ثانٍ بعد نجاح كلمة المرور، مثل:

- تطبيق Authenticator.
- رمز OTP.
- رمز يرسل عبر البريد الإلكتروني.

### الخيار الأفضل

يفضل استخدام تطبيق Authenticator المبني على TOTP بدل الاعتماد الكامل على البريد الإلكتروني.

### الفوائد

- تقليل خطر استخدام كلمات المرور المسروقة.
- رفع مستوى حماية الحسابات.
- تقليل أثر هجمات التصيد.
- إضافة حماية مناسبة لطبيعة النظام المالي.

### متطلبات الأمان

- عدم تخزين الرمز بصورته الأصلية عند الإمكان.

## 6. استعادة كلمة المرور

### التحسين المقترح

إضافة وظيفة:

```text
Forgot Password
```

### تدفق العمل المقترح

1. يدخل المستخدم بريده الإلكتروني.
2. ينشئ النظام رمزًا مؤقتًا.
3. يرسل رابط إعادة تعيين.
4. يتحقق النظام من صلاحية الرمز.
5. يطلب كلمة مرور جديدة.
6. يخزن Hash جديدًا.
7. يلغي جميع الجلسات السابقة عند الحاجة.

### ضوابط الأمان

- عدم كشف ما إذا كان البريد موجودًا.
- تحديد صلاحية زمنية للرمز.
- استخدام رمز لمرة واحدة.
- تقييد عدد الطلبات.
- تسجيل الحدث داخل Audit Log.

## 7. قفل الحساب المؤقت

يطبق المشروع Rate Limiting على تسجيل الدخول، لكن يمكن إضافة حماية على مستوى الحساب.

### التحسين المقترح

قفل الحساب مؤقتًا بعد عدد محدد من المحاولات الفاشلة.

### مثال

```text
5 محاولات فاشلة خلال 15 دقيقة
```

### الفوائد

- تقليل Brute Force.
- حماية الحساب حتى عند تغيير عنوان IP.
- دعم التنبيه الأمني للمستخدم.

### ملاحظة

يجب تصميم القفل بحذر حتى لا يستخدمه المهاجم في تعطيل حسابات الآخرين.

## 8. إضافة Audit Logging شامل

- تسجيل الدخول الفاشل.
- تسجيل الخروج.
- تغيير كلمة المرور.
- تغيير البريد الإلكتروني.
- إنشاء حساب.
- تعطيل حساب.
- إعادة تفعيل حساب.
- دخول لوحة الإدارة.
- التحويل الناجح.
- التحويل المرفوض.
- محاولة تجاوز الصلاحيات.
- تجاوز Rate Limiting.
- أخطاء الخادم المهمة.

### الحقول المقترحة

- نوع الحدث.
- رقم المستخدم.
- عنوان IP.
- User-Agent.
- التاريخ والوقت.
- نتيجة الحدث.
- وصف مختصر.
- مستوى الخطورة.

### بيانات يجب عدم تسجيلها

- كلمة المرور.
- Session Cookie.
- CSRF Token.
- SECRET_KEY.
- رموز إعادة تعيين كلمة المرور.
- معلومات مالية غير ضرورية.

### الفوائد

```text

```

بعد التأكد من تشغيل الموقع بالكامل عبر HTTPS.
