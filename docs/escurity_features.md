# الخصائص الأمنية في النظام المصرفي الآمن

## 1. مقدمة

يتعامل النظام المصرفي مع بيانات حساسة، مثل أسماء المستخدمين، كلمات المرور، أرقام الحسابات، الأرصدة، وسجلات التحويلات المالية.

لذلك تم بناء المشروع وفق مجموعة من مبادئ تطوير البرمجيات الآمنة، بهدف تقليل مخاطر الوصول غير المصرح به، سرقة كلمات المرور، التلاعب بالطلبات، تنفيذ تحويلات غير صحيحة، أو الوصول إلى وظائف المدير بواسطة مستخدم عادي.

لا يعتمد أمان النظام على إجراء واحد فقط، بل يستخدم عدة طبقات أمنية تعمل معًا، مثل:

- تشفير كلمات المرور.
- التحقق من المدخلات.
- حماية النماذج.
- المصادقة.
- التحكم في الصلاحيات.
- تحديد عدد المحاولات.
- حماية العمليات المالية.
- التراجع عن العمليات عند حدوث خطأ.
- صفحات أخطاء مخصصة.
- اختبارات أمنية آلية.

---

## 2. أهداف الأمان

تهدف الضوابط الأمنية في المشروع إلى تحقيق ما يلي:

1. حماية كلمات مرور المستخدمين.
2. منع الوصول إلى الصفحات الحساسة دون تسجيل الدخول.
3. منع المستخدم العادي من الوصول إلى لوحة المدير.
4. منع المستخدم المعطل من تسجيل الدخول.
5. منع إدخال بيانات غير صحيحة.
6. منع التحويلات المالية غير الصالحة.
7. منع الطلبات المزورة عبر النماذج.
8. تقليل محاولات التخمين والهجمات المتكررة.
9. ضمان سلامة تحديث الأرصدة.
10. منع حفظ بيانات جزئية عند حدوث خطأ.
11. حماية الملفات الحساسة من الرفع إلى GitHub.
12. التحقق من الضوابط الأمنية باستخدام الاختبارات.

## 3. نموذج التهديدات

يمكن أن يتعرض النظام للتهديدات التالية:

| التهديد | المثال | الإجراء الأمني |
|---|---|---|
| سرقة كلمات المرور | الوصول إلى قاعدة البيانات | تشفير كلمات المرور باستخدام Bcrypt |
| التخمين المتكرر | تجربة كلمات مرور كثيرة | Rate Limiting |
| الوصول غير المصرح | فتح Dashboard دون تسجيل الدخول | Flask-Login وlogin_required |
| تجاوز الصلاحيات | مستخدم عادي يفتح /admin | التحقق من الدور و403 |
| CSRF | إرسال تحويل باسم مستخدم آخر | Flask-WTF وCSRF Token |
| التلاعب بالمبلغ | إدخال مبلغ سالب أو صفر | التحقق من مبلغ التحويل |
| التحويل للحساب نفسه | استخدام رقم حساب المرسل | مقارنة الحسابين |
| تجاوز الرصيد | تحويل مبلغ أكبر من الرصيد | فحص الرصيد قبل التحويل |
| تكرار البيانات | استخدام بريد أو اسم مستخدم موجود | Unique Validation |
| حفظ بيانات ناقصة | فشل أثناء التحويل | Database Rollback |
| كشف معلومات المستخدم | إظهار أن الاسم موجود | رسالة دخول عامة |
| رفع بيانات حساسة | رفع قاعدة البيانات إلى GitHub | .gitignore |

## 4. تشفير كلمات المرور

لا يقوم النظام بحفظ كلمة المرور الأصلية داخل قاعدة البيانات.

بدلًا من ذلك، يتم تحويل كلمة المرور إلى قيمة مشفرة باستخدام:

Flask-Bcrypt
### إنشاء القيمة المشفرة

def hash_password(password: str) -> str:
    password_hash = bcrypt.generate_password_hash(password)
    return password_hash.decode("utf-8")
عند تسجيل المستخدم، يتم حفظ:

password_hash
بدلًا من كلمة المرور الأصلية.

### التحقق من كلمة المرور

عند تسجيل الدخول، تتم مقارنة كلمة المرور المدخلة بالقيمة المشفرة:

def check_password(
    password_hash: str,
    password: str
) -> bool:

    return bcrypt.check_password_hash(
        password_hash,
        password
    )
### أهمية Bcrypt

يوفر Bcrypt الخصائص التالية:

- لا يحفظ كلمة المرور الأصلية.
- يولد Salt مختلفًا لكل كلمة مرور.
- يجعل هجمات القاموس أكثر صعوبة.
- مصمم ليكون بطيئًا نسبيًا ضد محاولات التخمين.
- لا يمكن استعادة كلمة المرور من القيمة المشفرة مباشرة.

## 5. التحقق من قوة كلمة المرور

يتحقق النظام من قوة كلمة المرور قبل إنشاء الحساب أو تغييرها.

الشروط الحالية هي:

- 12 حرفًا على الأقل.
- حرف إنجليزي كبير.
- حرف إنجليزي صغير.
- رقم واحد على الأقل.
- رمز خاص واحد على الأقل.
- عدم وجود مسافات.

مثال على كلمة مرور مقبولة:

SecureUser@123
مثال على كلمة مرور مرفوضة:

password
دالة التحقق:

def validate_password(password: str) -> bool:

    if len(password) < 12:
        return False

    if any(character.isspace() for character in password):
        return False

    if re.search(r"[A-Z]", password) is None:
        return False

    if re.search(r"[a-z]", password) is None:
        return False

    if re.search(r"\d", password) is None:
        return False

    if re.search(r"[^\w\s]", password) is None:
        return False

    return True

## 6. حماية تسجيل الدخول
يتحقق النظام من:

1. وجود اسم المستخدم.
2. صحة كلمة المرور.
3. أن الحساب غير معطل.
4. صلاحية المستخدم.
5. الحد المسموح من المحاولات.

عند فشل تسجيل الدخول، يستخدم النظام رسالة عامة:

Invalid username or password.
لا يعرض النظام رسائل مختلفة مثل:

Username does not exist.
أو:

Password is incorrect.
يساعد ذلك على منع المهاجم من معرفة أسماء المستخدمين المسجلين.

is_active

True

if not user.is_active:

## 15. منع تكرار البيانات

قبل إنشاء مستخدم جديد، يتحقق النظام من عدم وجود:

- اسم مستخدم مطابق.

existing_email = User.query.filter_by(

username

## 16. التحقق من رقم الحساب

يتكون رقم الحساب من 12 خانة تحتوي على:

- الأرقام من 0 إلى 9.
- الحروف من A إلى F.

النمط المستخدم:

ACCOUNT_NUMBER_PATTERN = re.compile(
    r"^[A-F0-9]{12}$"
)
مثال صحيح:

37E8C71DF154
مثال غير صحيح:

123
قبل تنفيذ التحويل يتم التحقق من:

1. وجود رقم الحساب.
2. صحة تنسيقه.
3. وجود الحساب في قاعدة البيانات.
4. أن الحساب ليس حساب المرسل نفسه.

## 20. سلامة العملية المالية

يجب أن يتم:

- خصم المبلغ من المرسل.
- إضافة المبلغ إلى المستلم.
- إنشاء سجل العملية.

ضمن جلسة قاعدة بيانات واحدة.

عند النجاح:

db.session.commit()
عند حدوث خطأ:

db.session.rollback()
يساعد ذلك على منع حفظ عملية ناقصة.

---

## 21. تسجيل العمليات المالية

يتم إنشاء سجل لكل تحويل ناجح يحتوي على:

- رقم حساب المرسل.
- رقم حساب المستلم.
- قيمة التحويل.
- تاريخ ووقت العملية.

لا يتم إنشاء سجل للتحويلات المرفوضة.

يساعد سجل العمليات على:

- مراجعة التحويلات.
- معرفة المرسل والمستلم.
- تتبع الأرصدة.
- اكتشاف العمليات غير الطبيعية.
- توفير سجل للمراجعة.

## 22. تغيير كلمة المرور

عند تغيير كلمة المرور يقوم النظام بـ:

1. التحقق من كلمة المرور الحالية.
2. التحقق من تطابق كلمتي المرور الجديدتين.
3. التحقق من قوة كلمة المرور الجديدة.
4. منع استخدام كلمة المرور الحالية نفسها.
5. تشفير كلمة المرور الجديدة.
6. حفظها في قاعدة البيانات.
7. تسجيل خروج المستخدم.
8. طلب تسجيل الدخول بالكلمة الجديدة.

تسجيل خروج المستخدم بعد تغيير كلمة المرور يقلل مخاطر استمرار جلسة قديمة.

- صحة اسم المستخدم.

- عدم تكرار الاسم.

## 24. صفحات الأخطاء المخصصة

يحتوي النظام على صفحات أخطاء مخصصة.

### 403 — Access Denied

تظهر عند عدم امتلاك الصلاحية.

### 404 — Page Not Found

تظهر عند فتح رابط غير موجود.

### 429 — Too Many Requests

تظهر عند تجاوز عدد المحاولات.

### 500 — Internal Server Error

تظهر عند حدوث خطأ داخلي غير متوقع.

داخل معالج خطأ 500 يتم تنفيذ:

db.session.rollback()
لإلغاء أي عملية قاعدة بيانات غير مكتملة.

## 26. حماية المفتاح السري

.env

## 27. إعدادات الجلسة

يمكن تحسين حماية الجلسات باستخدام:

SESSION_COOKIE_HTTPONLY = True
SESSION_COOKIE_SAMESITE = "Lax"
عند استخدام HTTPS في بيئة الإنتاج يضاف:

SESSION_COOKIE_SECURE = True
### وظيفة الإعدادات

- HTTPOnly: يقلل وصول JavaScript إلى Cookie.
- SameSite: يقلل إرسال Cookie في الطلبات الخارجية.
- Secure: يسمح بإرسال Cookie عبر HTTPS فقط.

## 28. إيقاف وضع Debug في الإنتاج

أثناء التطوير يمكن تشغيل:

app.run(debug=True)
لكن عند نشر النظام يجب إيقاف Debug:

app.run(debug=False)
لأن صفحة Debug قد تكشف:

- مسارات الملفات.
- أسماء المتغيرات.
- تفاصيل قاعدة البيانات.
- أجزاء من كود النظام.
- معلومات داخلية حساسة.

## 31. الدفاع متعدد الطبقات

يعتمد المشروع على مفهوم:

Defense in Depth
أي استخدام أكثر من طبقة حماية.

مثال عملية تسجيل الدخول:

1. التحقق من الحقول.
2. البحث عن المستخدم.
3. مقارنة كلمة المرور المشفرة.
4. التحقق من حالة الحساب.
5. تطبيق Rate Limiting.
6. إنشاء جلسة للمستخدم.
7. التحكم في صلاحية الصفحة.

إذا فشلت إحدى الطبقات، تظل الطبقات الأخرى موجودة.

## 32. حدود الحماية الحالية

رغم الضوابط الموجودة، فإن المشروع تعليمي ولا يمثل نظامًا مصرفيًا إنتاجيًا كاملًا.

الوظائف غير الموجودة حاليًا:

- المصادقة الثنائية.
- التحقق من البريد الإلكتروني.
- استعادة كلمة المرور بالبريد.
- HTTPS المحلي.
- سجل أمني شامل.
- تنبيهات العمليات.
- تشفير قاعدة البيانات.
- مراقبة السلوك غير الطبيعي.
- انتهاء الجلسة بعد مدة محددة.
- موافقة إضافية على التحويلات الكبيرة.
- حماية من الأجهزة غير المعروفة.

## 33. تحسينات أمنية مستقبلية

يمكن تطوير النظام بإضافة:

### 33.1 المصادقة الثنائية

إرسال رمز مؤقت عبر:

- البريد الإلكتروني.
- تطبيق Authenticator.
- رسالة نصية.

### 33.2 سجل التدقيق

إضافة جدول:

audit_logs
يسجل:

- تسجيل الدخول.
- تسجيل الخروج.
- تغيير كلمة المرور.
- تعديل الملف الشخصي.
- عمليات المدير.
- التحويلات.
- المحاولات الفاشلة.

### 33.3 انتهاء الجلسة

إنهاء الجلسة بعد فترة من عدم النشاط.

### 33.4 التحقق من البريد

منع تفعيل الحساب قبل تأكيد البريد الإلكتروني.

### 33.5 تجميد الحساب

إضافة حالة للحساب البنكي:

active
frozen
closed
### 33.6 إشعارات التحويل

إرسال إشعار بعد كل عملية مالية.

### 33.7 قاعدة بيانات إنتاجية

استخدام PostgreSQL بدل SQLite عند النشر.

### 33.8 تشغيل HTTPS

استخدام شهادة TLS لحماية البيانات أثناء النقل.

### 33.9 مراقبة محاولات الدخول

اكتشاف:

- محاولات كثيرة من عنوان IP واحد.
- تسجيل الدخول من موقع جديد.
- محاولات متعددة على حسابات مختلفة.

## 34. الخاتمة

تم تطبيق عدة ضوابط أمنية داخل النظام المصرفي، تشمل تشفير كلمات المرور، حماية CSRF، المصادقة، التحكم في الصلاحيات، تحديد عدد المحاولات، التحقق من البيانات، حماية التحويلات، والتراجع عن العمليات عند حدوث خطأ.

تعمل هذه الإجراءات معًا ضمن مبدأ الدفاع متعدد الطبقات لتقليل المخاطر الأمنية.

كما تم اختبار الضوابط الأساسية باستخدام Pytest، ونجح المشروع في اجتياز 25 اختبارًا آليًا.

يبقى المشروع تعليميًا، ويمكن تطويره مستقبلًا بإضافة المصادقة الثنائية، سجل التدقيق، HTTPS، الإشعارات، والتحقق من البريد الإلكتروني.
