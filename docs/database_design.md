# تصميم قاعدة بيانات النظام المصرفي الآمن

## 1. مقدمة

تعتمد قاعدة بيانات النظام المصرفي الآمن على تنظيم البيانات في مجموعة من الجداول المترابطة، بهدف حفظ معلومات المستخدمين والحسابات البنكية والتحويلات المالية بطريقة واضحة ومنظمة.

تم استخدام SQLite كقاعدة بيانات أثناء مرحلة التطوير، مع استخدام مكتبة Flask-SQLAlchemy للتعامل مع الجداول والعلاقات من خلال كائنات Python.

تتكون قاعدة البيانات من ثلاثة جداول أساسية:

1. جدول المستخدمين users.
2. جدول الحسابات البنكية accounts.
3. جدول العمليات المالية transactions.

---

## 2. أهداف تصميم قاعدة البيانات

يهدف تصميم قاعدة البيانات إلى:

- حفظ بيانات المستخدمين بصورة منظمة.
- ربط كل مستخدم بحساب بنكي.
- حفظ رصيد كل حساب.
- تسجيل جميع التحويلات المالية.
- معرفة حساب المرسل والمستلم في كل عملية.
- منع تكرار اسم المستخدم والبريد الإلكتروني.
- منع تكرار أرقام الحسابات البنكية.
- ضمان سلامة العلاقات بين الجداول.
- تسهيل استرجاع البيانات وعرضها في النظام.

## 3. مخطط قاعدة البيانات

يوضح المخطط التالي العلاقة بين الجداول الرئيسية:

erDiagram

    USERS ||--o| ACCOUNTS : owns

    ACCOUNTS ||--o{ TRANSACTIONS : sends
    ACCOUNTS ||--o{ TRANSACTIONS : receives

    USERS {
        integer id PK
        string username UK
        string email UK
        string password_hash
        string role
        boolean is_active
        datetime created_at
    }

    ACCOUNTS {
        integer id PK
        string account_number UK
        decimal balance
        integer user_id FK
    }

    TRANSACTIONS {
        integer id PK
        integer sender_account_id FK
        integer receiver_account_id FK
        decimal amount
        datetime created_at
    }

## 4. جدول المستخدمين Users

يستخدم جدول users لحفظ بيانات المستخدمين المسجلين في النظام.

### اسم الجدول

users
### الحقول

| اسم الحقل | نوع البيانات | الوصف |
|---|---|---|
| id | Integer | الرقم الأساسي للمستخدم |
| username | String | اسم المستخدم |
| email | String | البريد الإلكتروني |
| password_hash | String | كلمة المرور بعد تشفيرها |
| role | String | صلاحية المستخدم |
| is_active | Boolean | حالة الحساب |
| created_at | DateTime | تاريخ إنشاء المستخدم |

### المفتاح الأساسي

id
يتميز كل مستخدم برقم فريد داخل قاعدة البيانات.

### الحقول الفريدة

يجب ألا تتكرر الحقول التالية:

username
email
وذلك لمنع إنشاء مستخدمين بنفس اسم المستخدم أو البريد الإلكتروني.

### حقل كلمة المرور

لا يتم حفظ كلمة المرور الأصلية داخل قاعدة البيانات.

بدلًا من ذلك، يتم حفظ القيمة المشفرة في:

password_hash
باستخدام Flask-Bcrypt.

### حقل الصلاحية

يخزن الحقل:

role
أحد النوعين:

customer
admin
- `customer: مستخدم عادي.
- admin: مدير النظام.

### حقل حالة المستخدم

يستخدم:

``text
is_active

لتحديد حالة الحساب:

- `True`: الحساب فعال.
- `False`: الحساب معطل.

## 12. آلية تنفيذ التحويل داخل قاعدة البيانات

عند تحويل مبلغ مالي، ينفذ النظام العمليات التالية:

1. البحث عن حساب المستخدم الحالي.
2. البحث عن حساب المستلم.
3. التحقق من توفر الرصيد.
4. خصم المبلغ من حساب المرسل.
5. إضافة المبلغ إلى حساب المستلم.
6. إنشاء سجل جديد في جدول العمليات.
7. حفظ جميع التغييرات معًا.

مثال:

text
رصيد المرسل قبل التحويل: 1000.00
قيمة التحويل: 100.00
رصيد المرسل بعد التحويل: 900.00


text
رصيد المستلم قبل التحويل: 1000.00
قيمة التحويل: 100.00
رصيد المستلم بعد التحويل: 1100.00

لا يتغير إجمالي الأموال داخل النظام:

text
1000 + 1000 = 2000

بعد التحويل:

text
900 + 1100 = 2000

## 13. سلامة العملية المالية

يتم تنفيذ تحديث الرصيدين وإنشاء سجل العملية ضمن جلسة واحدة لقاعدة البيانات.

عند نجاح جميع الخطوات يتم استخدام:

python
db.session.commit()

أما عند حدوث خطأ فيتم استخدام:

python
db.session.rollback()

يساعد ذلك على منع الحالات التالية:

- خصم المبلغ من المرسل دون إضافته للمستلم.
- إضافة المبلغ للمستلم دون خصمه من المرسل.
- تسجيل عملية مالية غير مكتملة.
- حفظ تغييرات جزئية داخل قاعدة البيانات.

## 14. القيود المستخدمة

تم استخدام عدة قيود لحماية البيانات.

### 14.1 Primary Key

يستخدم في كل جدول لتحديد كل سجل بصورة فريدة.

### 14.2 Foreign Key

يستخدم لربط الجداول ومنع إنشاء علاقات غير صحيحة.

### 14.3 Unique

يستخدم لمنع تكرار:

- اسم المستخدم.
- البريد الإلكتروني.
- رقم الحساب البنكي.

### 14.4 Not Null

يستخدم لمنع حفظ قيم فارغة في الحقول الأساسية.

### 14.5 Default Values

تستخدم لإضافة قيم افتراضية مثل:

text
role = customer
is_active = True
balance = 1000.00

## 18. سبب استخدام SQLite

تم اختيار SQLite للأسباب التالية:

- سهلة الاستخدام.
- لا تحتاج إلى خادم منفصل.
- مناسبة للمشاريع التعليمية.
- سهلة النسخ والنقل.
- متوافقة مع SQLAlchemy.
- مناسبة للاختبارات المحلية.

لكن في النظام الحقيقي يفضل استخدام:

- PostgreSQL.
- MySQL.
- Microsoft SQL Server.

## 20. ملاحظات على الحذف

في النسخة التعليمية يمكن للمدير حذف المستخدم التجريبي وحسابه والعمليات المرتبطة به.

لكن في الأنظمة المصرفية الحقيقية لا يفضل حذف السجلات المالية نهائيًا.

الأفضل هو:

- تعطيل المستخدم.
- الاحتفاظ بالعمليات.
- استخدام الحذف المنطقي.
- إضافة حقل مثل:

is_deleted
أو:

deleted_at
للحفاظ على السجلات لأغراض المراجعة والتدقيق.

## 22. الخاتمة

تم تصميم قاعدة بيانات النظام المصرفي الآمن باستخدام ثلاثة جداول رئيسية مترابطة، وهي المستخدمون، الحسابات، والعمليات المالية.

يوفر هذا التصميم طريقة منظمة لحفظ معلومات المستخدمين والحسابات البنكية والتحويلات، مع تطبيق القيود والعلاقات اللازمة لضمان سلامة البيانات.

كما تم استخدام المعاملات وقاعدة بيانات اختبار منفصلة لتقليل الأخطاء وحماية بيانات المشروع أثناء التطوير والاختبار.
