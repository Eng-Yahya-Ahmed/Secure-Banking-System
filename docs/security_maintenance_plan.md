# خطة الصيانة الأمنية والتحديث الدوري

## 1. مقدمة

توضح هذه الوثيقة خطة الصيانة الأمنية لمشروع Secure Banking System بعد انتهاء مرحلة التطوير الأولية.

تهدف الخطة إلى المحافظة على أمان التطبيق واستقراره من خلال تحديث المكتبات، ومراجعة السجلات، وإعادة الاختبارات، وإنشاء النسخ الاحتياطية، ومعالجة الثغرات الجديدة.

تُعد الصيانة مرحلة مستمرة ضمن دورة حياة تطوير البرمجيات الآمنة S-SDLC، ولا تنتهي بمجرد نشر التطبيق أو تسليمه.

---

## 2. أهداف خطة الصيانة

تهدف الخطة إلى:

- المحافظة على تحديث مكتبات المشروع.
- اكتشاف الثغرات الأمنية الجديدة.
- إعادة تشغيل الاختبارات بعد كل تعديل.
- مراجعة تقارير Bandit وOWASP ZAP.
- حماية الأسرار وبيانات الإعداد.
- المحافظة على سلامة قاعدة البيانات.
- مراقبة محاولات الدخول والأنشطة الحساسة.
- الاستجابة للحوادث الأمنية.
- توثيق التعديلات والإصلاحات.
- منع إعادة ظهور الثغرات التي تمت معالجتها.

## 4. جدول الصيانة الدورية

| المهمة | التكرار | المسؤول | النتيجة المطلوبة |
|---|---|---|---|
| تشغيل اختبارات Pytest | بعد كل تعديل | المطور | نجاح جميع الاختبارات |
| تشغيل Bandit SAST | بعد تعديل كود Python | المطور | عدم وجود مشكلات أمنية مهمة |
| تشغيل OWASP ZAP | بعد تغييرات الواجهات أو الإعدادات | فريق الاختبار | مراجعة التنبيهات ومعالجتها |
| مراجعة المكتبات | شهريًا | المطور | تحديث المكتبات الضعيفة أو القديمة |
| فحص الثغرات في المكتبات | شهريًا | المطور | عدم وجود ثغرات معروفة خطيرة |
| مراجعة إعدادات الأمان | كل ثلاثة أشهر | المسؤول الأمني | التأكد من سلامة الإعدادات |
| مراجعة صلاحيات المستخدمين | شهريًا | المدير | إزالة الصلاحيات غير الضرورية |
| مراجعة السجلات الأمنية | أسبوعيًا | المسؤول الأمني | اكتشاف الأنشطة المشبوهة |
| إنشاء نسخة احتياطية | يوميًا في الإنتاج | مسؤول النظام | نسخة سليمة قابلة للاستعادة |
| اختبار استعادة النسخة الاحتياطية | شهريًا | مسؤول النظام | التأكد من نجاح الاستعادة |
| تغيير الأسرار عند الاشتباه | عند الحاجة | مسؤول النظام | إلغاء السر القديم وإنشاء بديل |
| مراجعة التوثيق | بعد كل إصدار | المطور | توثيق مطابق للنسخة الحالية |

## 5. تحديث مكتبات المشروع

يجب مراجعة مكتبات Python الموجودة داخل:

```text
backend/requirements.txt
```

يمكن معرفة المكتبات القديمة باستخدام:

```powershell
pip list --outdated
```

بعد تحديث أي مكتبة يجب:

1. قراءة ملاحظات الإصدار.
2. التأكد من عدم وجود تغييرات غير متوافقة.
3. تحديث المكتبة داخل البيئة الافتراضية.
4. تحديث `requirements.txt`.
5. تشغيل جميع اختبارات Pytest.
6. تشغيل Bandit.
7. تشغيل التطبيق داخل Docker.
8. إعادة فحص OWASP ZAP عند الحاجة.
9. توثيق التحديث في Git.

لا يجب تحديث جميع المكتبات دفعة واحدة دون اختبار؛ لأن ذلك قد يصعب تحديد سبب المشكلة.

---

## 6. فحص الثغرات في المكتبات

يمكن مستقبلًا استخدام أدوات فحص الاعتماديات مثل:

```text
pip-audit
```

وظيفتها اكتشاف المكتبات التي تحتوي على ثغرات منشورة ومعروفة.

الأمر المقترح:

```powershell
pip-audit -r backend/requirements.txt
```

عند ظهور ثغرة يجب:

- تحديد المكتبة المتأثرة.
- قراءة مستوى الخطورة.
- البحث عن إصدار آمن.
- تحديث المكتبة.
- إعادة الاختبارات.
- توثيق المعالجة.

لا يكفي نجاح Bandit؛ لأن Bandit يفحص كود Python، بينما فحص الاعتماديات يركز على المكتبات الخارجية.

---

## 7. الاختبارات البرمجية

يجب تشغيل الاختبارات بعد كل تعديل مهم:

```powershell
python -m pytest tests -v
```

النتيجة الحالية المتوقعة:

```text
25 passed
```

عند ظهور اختبار فاشل يجب:

1. عدم رفع التعديل مباشرة.
2. قراءة رسالة الخطأ.
3. تحديد الملف أو الوظيفة المتأثرة.
4. إصلاح المشكلة.
5. إعادة جميع الاختبارات.
6. التأكد من نجاح GitHub Actions.

يجب إضافة اختبارات جديدة عند إضافة وظيفة جديدة أو معالجة ثغرة أمنية.

---

## 8. فحص SAST باستخدام Bandit

يتم تشغيل Bandit بعد تعديلات كود Python:

```powershell
python -m bandit -r backend -x backend/venv,backend/instance
```

ويجب حفظ تقرير الفحص داخل:

```text
docs/security_reports/
```

عند ظهور تنبيه يجب مراجعته وعدم تجاهله مباشرة.

يجب استخدام `# nosec` فقط عندما:

- يكون التنبيه معروفًا ومفهومًا.
- تكون هناك حاجة تقنية واضحة.
- يتم تحديد رقم القاعدة.
- تتم إضافة تعليق يشرح السبب.

مثال:

```python
host="0.0.0.0",  # nosec B104
```

ويجب عدم استخدام `# nosec` لإخفاء مشكلة أمنية حقيقية.

## 9. فحص DAST باستخدام OWASP ZAP

يجب إعادة فحص التطبيق عند تعديل:

- نماذج الدخول أو التسجيل.
- صفحات التحويل.
- إعدادات Cookies.
- رؤوس الحماية.
- سياسة CSP.
- ملفات JavaScript أو Bootstrap.
- صلاحيات الوصول.
- إعدادات Docker أو الخادم.

الفحص الحالي هو:

```text
OWASP ZAP Baseline Scan
```

وقد شمل الصفحات العامة دون تسجيل دخول.

التحسين المستقبلي هو تنفيذ فحص مصادق عليه يشمل:

- Dashboard.
- Transfer.
- Transactions.
- Profile.
- Change Password.
- Admin Panel.

بعد أي إصلاح يجب إعادة الفحص للتأكد من اختفاء التنبيه وعدم ظهور مشكلات جديدة.

- مفاتيح خدمات البريد.

1. إلغاء السر القديم فورًا.
2. إنشاء قيمة جديدة.
3. تحديث بيئة التشغيل.
4. مراجعة السجلات.
5. التحقق من عدم إساءة استخدام السر.

## 12. النسخ الاحتياطية

في البيئة الحالية تُستخدم قاعدة بيانات SQLite.

في بيئة الإنتاج يجب إنشاء نسخ احتياطية دورية لقاعدة البيانات.

يجب أن تكون النسخ الاحتياطية:

- مشفرة.
- محفوظة في موقع منفصل.
- محددة الصلاحيات.
- مرتبطة بتاريخ ووقت.
- قابلة للاستعادة.
- محمية من الحذف غير المصرح.

لا يكفي إنشاء النسخة الاحتياطية؛ يجب اختبار عملية الاستعادة دوريًا.

### خطة مقترحة

- نسخة يومية.
- الاحتفاظ بالنسخ اليومية لمدة سبعة أيام.
- نسخة أسبوعية.
- الاحتفاظ بالنسخ الأسبوعية لمدة شهر.
- نسخة شهرية طويلة المدى.
- اختبار الاستعادة مرة كل شهر.

## 14. مراجعة الصلاحيات

يجب تطبيق مبدأ:

```text
Least Privilege
```

أي منح كل مستخدم الحد الأدنى فقط من الصلاحيات التي يحتاجها.

يجب مراجعة:

- المستخدمين الذين لديهم دور Admin.
- الحسابات غير النشطة.
- الحسابات القديمة.
- صلاحيات الوصول إلى التقارير.
- صلاحيات الوصول إلى قاعدة البيانات.
- صلاحيات ملفات النسخ الاحتياطية.

عند مغادرة مستخدم أو انتهاء حاجته للصلاحية، يجب إلغاء الوصول فورًا.

## 23. الخلاصة

تساعد هذه الإجراءات على تقليل المخاطر والمحافظة على أمان النظام واستقراره مع استمرار تطويره.
